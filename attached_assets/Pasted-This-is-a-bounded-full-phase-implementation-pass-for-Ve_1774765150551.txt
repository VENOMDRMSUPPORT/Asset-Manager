## Implementation requirements
Implement this in a provider-agnostic architecture with a current Z.AI-backed execution path.

Required design rules:

- Create a proper visual task pipeline inside VenomGPT rather than scattering image handling across ad hoc prompt logic.
- Separate:
  - task intake classification
  - capability detection
  - route selection
  - visual execution
  - result normalization
  - honest failure handling
- Keep the user experience simple:
  the user submits a screenshot/image task and the system handles routing automatically in the background.

### Routing and speed requirements
- Speed is critical for screenshot-task routing.
- Auto model selection must be fast and lightweight.
- Do not perform heavy runtime probing for every task.
- Prefer a cached or config-driven capability registry / route policy.
- Build a route selection mechanism that can quickly determine whether a true visual path is available.

### Current Z.AI-backed implementation requirement
Use Z.AI as the current concrete backend, but do not architect the system as “Z.AI-only.”
Use real vision-capable Z.AI models for screenshot/image analysis.
Prefer a fast vision-capable model as the primary route for speed-sensitive screenshot tasks, and allow a stronger fallback route when needed for harder visual analysis.

Do not assume all Z.AI models support images.
Do not send screenshot tasks into non-vision text models and pretend multimodal support exists.

### MCP policy
- Z.AI MCP servers may be used only as optional bounded enrichment if they materially improve the exact screenshot-analysis workflow.
- MCP must not be the main or only architecture path.
- The core screenshot-analysis flow must still make sense if a future provider is added without Z.AI MCP support.
- Do not tightly couple core task routing to provider-specific MCP surfaces.

### UI / lifecycle requirements
- Add only the minimum UI needed for image/screenshot task intake and result display.
- Preserve the current operator-facing lifecycle clarity.
- Ensure screenshot tasks have clear task state, logs, and output visibility.
- Do not redesign the whole interface.

### Analysis quality requirements
The screenshot-analysis output must be useful for real coding/debugging work, not generic image captioning.
Focus especially on frontend/UI issue detection and precise design observations.

The assistant should be able to return findings such as:
- what looks visually wrong
- where the likely UI issue is
- what component or region appears affected
- whether the problem looks like CSS/layout/responsive/state/rendering behavior
- what the likely implementation direction is

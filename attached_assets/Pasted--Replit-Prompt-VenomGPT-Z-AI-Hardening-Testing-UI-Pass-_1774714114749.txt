## What actually happened

We validated the system locally and found a very specific problem class:

- the model sometimes returns **multiple JSON objects in one response**
- the model sometimes mixes:
  - `think`
  - executable actions
  - `done`
- the current parsing / response-contract handling is too brittle
- this causes false failures such as:
  - “model returned invalid responses repeatedly”
- this can happen even on simple prompts like `hi`

At the same time, local validation also proved the core system is **not fundamentally broken**:
- single file tasks succeed
- multi-step file tasks succeed
- command execution succeeds
- verification succeeds

So this is **not** a rewrite request.

This is a **focused hardening pass** for:
1. the boundary between model output and action execution
2. Z.AI provider architecture and model-selection policy
3. a small local testing UI usability pass that is necessary for efficient validation

## Strategic direction

- select models automatically and quickly

- log what model was chosen and why

Also, the local testing UI currently slows down validation.  
We need a **small but practical usability pass** so repeated testing and debugging are not painful.

# Required outcomes

## 1) Add a response normalization layer before final parse/validation

The system must become more tolerant of common real-world model outputs, including cases such as:

- a single valid JSON object
- multiple JSON objects separated by newlines
- JSON wrapped in code fences
- a valid action plus extra explanatory text
- `think` followed by a valid executable action
- `think` followed by `done`
- trailing prose after an otherwise valid action object

Normalization must be conservative and safe.

Rules:
- do not silently invent actions
- do not guess destructively
- do recover from common formatting noise when recovery is unambiguous
- do preserve honest failure when recovery is not safe

## 2) Add explicit repair / retry behavior for imperfect structured output

The system must **not fail immediately** on imperfect model output.

Implement a clear recovery policy such as:
- normalize first
- attempt safe extraction
- validate extracted candidate actions
- retry with a stricter response instruction when needed
- optionally reuse safely extracted valid action content when unambiguous
- only declare failure after recovery attempts are exhausted

Important:
- do not create fake actions
- do not hide ambiguity
- do not claim success when the model output still cannot be trusted

A guiding rule for this pass:

> The system must not fail immediately on imperfect model output.  
> It must attempt normalization, extraction, and safe recovery before declaring failure.

## 3) Improve action parsing and validation

After normalization, validate actions with a clear contract.

The parser must:
- distinguish malformed JSON from multiple-object output
- distinguish unsupported action types from formatting failures
- distinguish conversational / non-task output from action output
- surface specific failure reasons in logs
- avoid collapsing unrelated failures into one vague “invalid response” bucket

## 4) Handle non-execution prompts correctly

Simple conversational prompts like:
- hi
- hello
- thanks
- ok

must not trigger brittle agent failures.

Implement a clean policy for prompts that do not require file operations or command execution.

Valid approaches include:
- bypassing agent execution for clearly non-execution prompts
- or allowing clean `done`-only completion in the agent loop

Choose the least disruptive architecture-consistent solution.

## 5) Refactor provider naming and provider boundaries

The Z.AI integration must have a clear provider identity and naming convention now, before more providers are added later.

Requirements:
- use a clear provider name consistently in code, config, logs, and diagnostics
- avoid vague naming that will become confusing in multi-provider phases
- keep provider-specific concerns isolated from generic orchestration logic
- keep provider diagnostics explicit and readable

## 6) Replace hardcoded model dependence with automatic fast backend model selection

Do not keep the system tightly coupled to one hardcoded main coding model in `.env`.

We want **fast automatic model selection in the backend**.

Requirements:
- remove the need to hardcode the main coding model in `.env`
- keep override ability for debugging, emergency pinning, or experiments
- implement model selection policy that quickly chooses an appropriate Z.AI model based on task type / capability need
- log the selected model and the reason for selection
- preserve safe fallback behavior if a preferred model is unavailable or unsupported

The goal is:
- the user should not need to manually update `.env` every time Z.AI releases a stronger model
- the backend should make a good default decision automatically

## 7) Support the broader Z.AI family at the provider layer

The provider layer must stop being artificially narrow.

Refactor it so it can represent and route cleanly across Z.AI capability categories such as:
- coding / language models
- vision models
- image models
- video models
- audio models
- tool-capable APIs
- agent-related APIs

Important:
- this does **not** require fully productizing all of these right now
- it **does** require a future-ready provider/capability structure now
- unsupported capabilities must fail honestly and clearly

## 8) Build a provider capability registry or equivalent mapping

Implement a clean internal structure that makes it easy to answer:
- which Z.AI capabilities exist
- which model/API supports which capability
- which calls are synchronous vs asynchronous
- which capabilities are implemented now vs deferred
- which route should be used for text, vision, image, video, audio, tools, or agents

This can be a registry, manifest, mapping, policy object, or similar architecture-consistent mechanism.

It must be easy to extend later.

## 9) Prepare clean extension points for Z.AI tools and agents

Z.AI exposes more than plain chat completion.

Prepare the code structure so future phases can integrate things such as:
- web search
- tool streaming output
- function calling
- structured output
- agent APIs

Do not overbuild.
Do not ship fake support.
But do refactor the provider layer so these are first-class extension points rather than hacks later.

## 10) Improve Z.AI error handling using provider semantics

Audit current Z.AI API error handling and improve it.

Requirements:
- classify HTTP-level failures separately from business error code failures
- surface useful user-safe failure summaries
- keep detailed provider diagnostics in logs
- handle rate limiting, auth/config problems, unsupported model or method, quota / balance issues, provider unavailable, and similar failures more explicitly
- avoid generic “internal error” messaging when a clearer classification is possible

## 11) Make a focused usability pass on the local testing UI

This phase is still primarily a backend/agent hardening pass, but a **small, targeted UI usability pass is required** because the current testing workflow is unnecessarily painful.

This is **not** a redesign request.  
It is a **practical testing UX pass** so the product can be validated properly.

- keep workspace boundaries safe

### F) Preserve developer usability
The UI should stay simple and local-dev friendly.

Do not overdesign it.  
Do not add product fluff.  
Do add the minimal interaction quality needed for repeated validation sessions.

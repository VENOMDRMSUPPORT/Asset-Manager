* file creation and modification tasks

The next highest-value gap is this:

VenomGPT still does not have a strong, trustworthy multimodal intake path for screenshot-driven and image-based coding tasks.

Right now, the system is materially stronger on text/coding tasks than on tasks that begin from visual evidence such as:

* a frontend screenshot
* a UI defect screenshot
* a layout/alignment problem image
* a screenshot of an application error state
* an image plus a short instruction like “fix this UI issue”

Visual Analysis + Rich Multimodal Task Intake

What this phase is trying to achieve

This phase should make VenomGPT materially better at handling coding-related tasks whose primary input is visual, especially screenshots.

The goal is not generic image captioning.

The goal is not image generation.

The goal is not broad multimedia expansion.

The goal is to make screenshot-driven and image-assisted coding tasks enter the system correctly, route correctly, and produce grounded, useful analysis for debugging, UI inspection, and task planning.

Required outcomes

1. Add or improve multimodal task intake so the system can recognize and structure tasks that include screenshots or images, not only plain text.

2. Distinguish visual task types in a useful way, such as:

   * screenshot-based UI issue
   * frontend/layout defect
   * image-assisted debugging task
   * mixed text + image coding request

3. Ensure routing/capability selection for these tasks is provider-aware and capability-aware, rather than brittle or hardcoded in a way that assumes one provider path always supports vision.

4. Produce useful visual-analysis outputs for coding work, not generic image descriptions. The system should be able to surface things like:

   * likely visible UI defects
   * layout/alignment problems
   * suspicious states visible in the screenshot
   * ambiguity where the image is insufficient
   * likely code areas or component areas to inspect next

5. Preserve the current execution-oriented architecture and working text/coding paths. This phase must extend the system, not destabilize what already works.

6. Improve result honesty. The system must clearly separate:

   * what is visibly present in the image
   * what is inferred
   * what still needs repo inspection or execution validation

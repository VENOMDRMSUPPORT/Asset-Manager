# Part 1 — Parse/Recovery Tightening

## Goal
Reduce visible malformed-response failures and make recovery quieter and more robust.

## Requirements
- improve malformed JSON/action extraction when model output is close to valid
- reduce noisy visible parse failure events when recovery succeeds immediately afterward
- preserve honest classification when recovery truly fails
- do not fake success
- do not remove the recovery/evidence discipline already added

We want:
- fewer “attempt 1/3 malformed” moments on normal tasks
- better first-pass resilience
- same honesty when the model is actually unusable

# Part 3 — Historical Persistence / Replay Improvement

## Goal
Bring task replay/history closer to a professional operator experience.

## Requirements
- completed task details should remain reopenable more reliably
- preserve or rehydrate useful task execution details after normal interaction
- improve persistence/replay beyond fragile in-memory-only behavior where practical
- if full backend persistence is too large for this pass, implement the strongest bounded version that materially improves task-history usability
- clearly distinguish:
  - active task
  - selected/viewing historical task
  - completed task summary
  - replayable details/logs

We want the user to be able to inspect prior work in a serious way, not lose it too easily.

# Part 4 — Operator-Facing Execution UI Polish

## Goal
Make the runtime experience feel more like a professional execution console / asset manager, closer in spirit to Replit.

This is NOT a request for cosmetic redesign for its own sake.
This is a request for a more useful operator-facing execution surface.

## Requirements

### A) Left-side execution feed / asset-manager feel
Add or improve a dedicated operator-facing execution feed that shows what the assistant is doing in a structured, professional way.

Desired behavior:
- grouped execution updates
- meaningful action summaries
- visible progress through stages
- compact but informative updates
- professional completion summary after task completion

Examples:
- inspecting relevant files
- editing target file
- running verification command
- repairing after failed verification
- restarting service
- clean typecheck
- completed with evidence

This feed must be grounded in real execution events, not fake narration.

### B) Right-side task context / task management polish
Improve the right-side task area so it feels more professional and updates live.

Requirements:
- clearer task list updates
- better visibility of current/selected task
- strong live-updating task state
- better post-completion summary display
- if useful, show task-relevant files touched / commands run / evidence summary rather than duplicating the entire left file tree

Important:
Do not create redundant duplicate file explorers just for looks.
Prefer task context and execution relevance over duplication.

### C) Summary quality
After a task completes, show a clean professional summary that helps the user trust the result:
- files changed
- commands run
- verification outcome
- repair attempts if any
- final status
